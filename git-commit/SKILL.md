# Git コミット

このスキルは、リポジトリ内の変更を確認し、関連する変更だけを stage し、内容に合うコミットメッセージで commit するために使います。

## 基本方針

- ユーザーが「コミットして」と明示した場合だけ commit する。
- 変更内容を確認せずに stage や commit をしない。
- ユーザーや他の作業者による無関係な変更を勝手に戻さない。
- stage するのは、依頼内容と関係するファイルだけにする。
- コミットメッセージは差分から判断し、短く具体的にする。

## 手順

1. 作業ツリーの状態を確認する。

```bash
git status --short
git diff --stat
git diff --name-only
```

2. 変更内容を読む。

```bash
git diff -- <file>
```

複数ファイルがある場合は、変更のまとまりを見極める。無関係な変更が混ざっている場合は、依頼された作業に関係するファイルだけを対象にする。

3. 必要なら確認コマンドを実行する。

プロジェクトに test、build、lint などの確認手段が明確にある場合は、リスクに応じて実行する。実行できなかった場合や省略した場合は、その理由を報告する。

4. コミット対象を stage する。

```bash
git add <file1> <file2>
```

`git add .` は、変更がすべて依頼内容に関係すると確認できる場合だけ使う。迷う場合はファイルを明示して stage する。

5. stage 済み差分を確認する。

```bash
git diff --cached --stat
git diff --cached --name-only
```

必要に応じて `git diff --cached` で中身も確認する。

6. コミットメッセージを決める。

メッセージは変更内容から自然に判断する。Conventional Commits が合う場合は使う。

例:

- `fix: correct task completion handling`
- `feat: add settings import flow`
- `chore: update release metadata`
- `docs: clarify setup steps`
- `refactor: simplify sync logic`

7. commit する。

```bash
git commit -m "<message>"
```

## レビュー観点

コミット前に以下を確認する。

- 変更ファイルが依頼内容と一致しているか
- 不要な生成物、ログ、ローカル設定ファイルが含まれていないか
- 秘密情報やトークンらしき文字列が追加されていないか
- 影響範囲が大きい変更を説明できるか
- 確認コマンドの結果が成功しているか、または未実行理由が明確か

## 報告

最終報告では以下を簡潔に伝える。

- コミットハッシュ
- コミットメッセージ
- 主な変更内容
- 実行した確認コマンドと結果
- 作業ツリーが clean かどうか

問題がある場合は、コミット結果より先に問題点を伝える。
