# concept-documentizer

本スキルは、現在のセッション内での対話、調査内容、ソースコードの解析結果から、「学んだことの全体像」を論理的かつ体系的な概念ドキュメント（Markdown）として `docs/` ディレクトリに書き出すための手順です。

## 核心的な原則
1. **詳細（具体的なコードの行や細かな変数名など）はあえて書かない**: 
   - ソースコードそのものがドキュメントの代わりになるような情報は記述をスキップするか、抽象化してください。
   - 目的は「パッと見て全体の概念やアーキテクチャ、仕組みが理解できること」です。
2. **体系的な構造**:
   - 概要、採用している方式/設計、全体の概念フロー、安全性や効率性などの利点/考察、という綺麗なレイアウトでまとめます。
3. **明示的な指示でのみ実行する**:
   - ユーザーから「セッションのまとめをドキュメント化して」という明確な指令（Directive）があった場合のみ起動します。

## 実行手順

### 1. セッション履歴とコードのコンテキスト確認
- セッション内で議論された内容や、調査したファイルを振り返ります。
- 開発に直接必要なソースコード上の知識ではなく、「背後にある仕組み・概念」が何であるかを抽出します。
- 必要であれば関連するソースコードを調査する。

### 2. ドキュメントの構成設計
Markdownファイルは以下の標準構成をベースに、セッション内容に合わせて調整します。
```markdown
# [対象システム名・セッションテーマ] 概要

本セッションを通じて整理・理解した[システム/機能]の仕組みについて。

## 採用方式・設計パターン
- どのような技術、プロトコル、設計が採用されているか

## 全体概念フロー
1. [ステップ1]
2. [ステップ2]
3. ...

## 利点・安全性・効率性について
- この方式を採用することによるメリットや、計算された美しさ、安全性についての考察
```

### 3. ファイルの出力
- 成果物は必ずプロジェクトの `docs/` ディレクトリ配下に、適切な名前（例: `session_summary.md` や `auth_overview.md`）で保存します。
- ディレクトリが存在しない場合は自動で作成します。

## 出力例
ユーザーへの報告時は、以下のように簡潔かつ論理的に成果を伝えてください。
「ご指示通り、セッションの学びを体系的にまとめた概念ドキュメントを `docs/xxx.md` として作成しました。詳細なコードには深入りせず、全体像を効率的に把握できるように整理してあります。」
